System ideas
The stack should explain itself in trading terms, not just model terms.
Technology is only useful here if it changes expected trade behavior. These notes describe the current QUANTA architecture in that spirit.
Architecture
Thor-only single specialist operating path
Entry

Impulse gating, then score ranking

Thor is designed to ignore most bars and care about the small subset that already looks like an impulse breakout. The relevant question is not “is price up?” but “is this the class of event Thor was trained to exploit?”

Gate stack: pre_impulse_r2 <= 0.70 UTC hour not in {0, 7, 10, 11, 12} Thor score >= 68 wave_strength >= 40
Exit

Bank once, then let the runner prove it deserves to live

The current exit stack is built around one partial bank and one runner. The runner is supposed to die on state, not on a cosmetic calendar clock.

Thor v2: SL = 3.0 ATR bank = dynamic Gompertz target bank fraction = 35% trail activate = 1.5 ATR above bank runner trail = 2.0 ATR
Parity

Paper execution now follows the walk-forward model

The paper path was forced to behave like the WF sim instead of mixing in extra live execution logic. That means bar-based progression, sim-style fills, and no live-only risk blockers on Thor paper entries.

Parity points: single synthetic fill bar-count timing closed 5m bar progression sim-style pyramid logic sim-style cooldown timestamps
Supervision

Runtime quality is part of the strategy

A high-quality model on a degraded runtime is still a low-quality system. Queue depth, feed health, drawdown guards, and calibration monitoring are all part of whether Thor is actually trustworthy live.

Monitors surfaced here: rolling accuracy calibration error feature drift risk events runtime log tail
